# QSM开发与服务分离设计

## 🎯 核心原则

**开发版本与服务版本完全分离**：
- 开发版本：随时开发、测试、迭代
- 服务版本：稳定运行、用户访问
- 升级流程：开发完成 → 测试通过 → 覆盖部署

## 📁 目录设计

```
开发版本位置：
/root/QSM/Web/              # Web操作系统开发版
/root/QSM/VM/               # 量子虚拟机开发版
/root/QSM/OS/               # 原生操作系统开发版

服务版本位置：
/var/www/som.top/           # Web服务版本（用户访问）
（其他服务版本待部署）
```

## 🔄 三种版本的升级流程

### Web操作系统（在线服务）

```
开发流程：
/root/QSM/Web/ 开发新功能
      ↓
本地测试验证
      ↓
确认稳定后执行升级脚本
      ↓
cp -r /root/QSM/Web/* /var/www/som.top/
      ↓
用户立即看到新版本
```

**升级频率**：每天晚上一次（或功能完成后）

### 量子虚拟机（应用安装）

```
开发流程：
/root/QSM/VM/ 开发新功能
      ↓
测试验证
      ↓
打包新版本（v1.0.1）
      ↓
发布到下载站/应用商店
      ↓
用户下载更新
```

**升级频率**：按版本号发布（v1.0.0 → v1.0.1 → v1.1.0）

### 原生操作系统（系统安装）

```
开发流程：
/root/QSM/OS/ 开发新功能
      ↓
完整测试
      ↓
打包系统镜像
      ↓
发布新版本
      ↓
用户更新系统
```

**升级频率**：按大版本发布（v1.0 → v2.0）

## 🚀 升级脚本

### Web版本升级脚本

```bash
#!/bin/bash
# 文件: /root/QSM/scripts/upgrade_web.sh

echo "🔄 开始升级Web服务..."

# 1. 备份当前版本
echo "📦 备份当前版本..."
cp -r /var/www/som.top /var/www/som.top.bak.$(date +%Y%m%d_%H%M%S)

# 2. 复制新版本
echo "📤 部署新版本..."
cp -r /root/QSM/Web/* /var/www/som.top/

# 3. 重启服务（如有需要）
# systemctl restart nginx

echo "✅ Web服务升级完成！"
echo "📊 新版本已部署到: /var/www/som.top/"
```

### 执行升级

```bash
# 每天晚上执行
cd /root/QSM
bash scripts/upgrade_web.sh
```

## 📊 工作流程

### 开发阶段（白天）
```
在 /root/QSM/Web/ 开发
    ↓
本地测试验证
    ↓
Git提交到开发分支
    ↓
不影响线上用户
```

### 升级阶段（晚上）
```
确认功能稳定
    ↓
执行升级脚本
    ↓
覆盖服务版本
    ↓
用户获得新功能
```

## 🎯 优势

1. **安全性**：开发不会影响正在使用的用户
2. **稳定性**：只有测试通过的版本才部署
3. **可回滚**：服务版本有备份，出问题可恢复
4. **高效性**：开发和服务完全分离，互不干扰

## 📋 日常流程

### 开发者（我）的工作
```
上午/下午：在 /root/QSM/Web/ 开发新功能
傍晚：测试新功能
晚上：确认稳定后执行升级脚本
```

### 用户体验
```
白天：正常使用稳定版本
晚上：自动获得新功能（无感知升级）
```
